# Phase 3.5: Meeting Minutes Upgrade Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade meeting-minutes skill from v1.0 (paste text → format) to v2.0 (audio → transcript → formal minutes in .md + .docx), supporting 特推會, IEP (4 variants + merged), and 行政會議.

**Architecture:** Single skill upgrade — meeting-minutes becomes the "meeting record manager" handling transcription (Groq Whisper), type detection, template application, and dual output (.md + .docx via pandoc). Templates use AI-GUIDE HTML comments to instruct LLM behavior per meeting type.

**Tech Stack:** Groq Whisper API (whisper-large-v3), pandoc, ffmpeg, textutil (macOS built-in)

**Spec:** `docs/plans/2026-03-16-phase3.5-meeting-minutes-upgrade-design.md`

### Task 10: Rewrite SKILL.md to v2.0

**Files:**
- Rewrite: `~/.openclaw/workspace/skills/meeting-minutes/SKILL.md`

- [ ] **Step 1: Write SKILL.md v2.0**

The new SKILL.md must contain ALL of the following sections (refer to spec "Skill 設計" section for details):

**Frontmatter:**
```yaml
---
name: meeting-minutes
description: "會議紀錄產生器 — 從錄音檔或文字素材產出正式會議紀錄（.md + .docx）。支援特推會、IEP（4 種時間點 + 合併會議）、行政會議。Trigger on: /meeting, '幫我做會議紀錄', '會議紀錄', 'meeting minutes', '處理會議錄音'."
---
```

**Sections to include:**

1. **Title & one-line description** — "會議紀錄產生器 v2.0"

2. **Supported Formats table** — 5 types (特推會, IEP-期初擬訂, IEP-期末檢討, IEP-合併會議, 行政會議) with template file references

3. **Settings block** — VAULT path, inbox path, transcripts path, Groq config path, Discord channel, mention

4. **Groq API Key reading** — bash snippet to read from groq-config.json

5. **Workflow (6 steps):**

   **Step 1: 掃描 inbox**
   - List files in inbox/ (.m4a, .mp3, .wav, .doc, .md, .txt)
   - .doc files: use `textutil -convert txt` to extract text
   - No files → ask user to paste text
   - One file → process directly
   - Multiple files → list and ask user which one(s)

   **Step 2: 判斷會議類型**
   - Present 5-option menu (特推會 / IEP 期初 / IEP 期末 / IEP 合併 / 行政會議)
   - Collect type-specific info (學年度, 第幾次, 學生姓名, etc.)

   **Step 3: 轉逐字稿（音檔時）**
   - Read GROQ_API_KEY from groq-config.json
   - Check file size; if > 25MB: ffmpeg compress → ffmpeg segment (20min chunks) → sequential Groq calls → merge
   - Groq Whisper API call: model=whisper-large-v3, language=zh, response_format=text
   - Save transcript to `transcripts/YYYY-MM-DD-<主題>.txt`
   - If text file or pasted text: skip this step

   **Step 4: 套範本產出 markdown**
   - Read matching template (with AI-GUIDE)
   - Fill template from transcript/text
   - Follow AI-GUIDE 用語指引
   - IEP 合併會議 → two-pass processing (spec lines 237-249):
     - Pass 1: template-iep-review.md → extract review content
     - Pass 2: template-iep-initial.md → extract planning content
     - If content is intertwined, mark `[待確認歸屬]`
   - Student name masking: IEP = full name, 特推會 = ○ mask
   - Add frontmatter (type, date, meeting-type, student tag)

   **Step 5: 使用者審稿**
   - Present draft in Discord
   - Iterate on feedback until approved

   **Step 6: 存檔**
   - Save .md to `meetings/YYYY-MM-DD-<主題>.md`
   - pandoc → .docx with matching reference doc:
     - IEP → `references/reference-iep.docx`
     - 特推會 → `references/reference-special-ed.docx`
     - 行政會議 → `references/reference-admin.docx`
   - Move audio to `inbox/done/`
   - Report completion with file list

6. **Naming Rules table** — per spec "輸出檔案命名規則"

7. **Frontmatter format** — per spec "輸出 markdown frontmatter"

8. **Error handling** — per spec "容錯" section:
   - Missing directories → auto-create
   - Groq API failure → fallback to manual paste
   - Unsupported audio → try ffmpeg convert
   - Poor transcript quality → mark `[語音辨識不清]`
   - pandoc missing → warn, output .md only

9. **Writing rules** — per spec "寫入規則":
   - Follow template structure exactly
   - Preserve professional terminology verbatim
   - Use `[待補]` for missing info, never fabricate
   - Decisions must be actionable
   - IEP = full names, 特推會 = ○ mask

10. **IEP 合併會議 detailed logic** — per spec lines 237-249
